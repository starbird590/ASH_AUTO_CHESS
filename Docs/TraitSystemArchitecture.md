# 羁绊系统架构草案

本文记录当前已确认的羁绊表、运行时结算和 UI 展示规则。后续实现导入器、运行时和界面时，以这里为准。

## 数据入口

Excel 是策划编辑源，不由运行时代码直接读取。

Unity 实际读取或导入的文件建议落在 `Assets/Data`：

- `Trait.csv`：由基础羁绊表导出。
- `TraitTier.csv`：由子羁绊表导出。

代码和文档中优先使用 `Trait` 这个领域词。表字段可以暂时保留 `UnionId` 兼容现有单位表，但新增代码不要继续扩大 `Union` 这个旧叫法。

子羁绊表里的 `Unionld` 应更正为 `UnionId`。这里的第三个字符应该是大写 `I`，不是小写 `l`。

## 表格职责

`Trait.csv` 定义羁绊本体：

- `UnionId`：羁绊唯一 ID，推荐使用字符串格式，例如 `U1001`。
- `UnionName`：羁绊名称。
- `UnionDes`：羁绊总描述。
- `UnionIcon`：羁绊图标名或路径。

`TraitTier.csv` 定义羁绊档位：

- `UnionChildId`：档位行唯一 ID，用于定位和调试，不参与激活计算。
- `UnionId`：所属羁绊 ID，需要和 `Trait.csv` 对应。
- `UnionCounts`：激活该档位需要的去重已部署玩家单位数量。
- `LevelDes`：给玩家看的档位描述，只用于 UI，不参与战斗逻辑。
- `SkillID`：该档位实际授予的技能 ID，支持 `{skillId1;skillId2}`。

## 导入与资产

导入后继续生成或更新 `TraitSO` 资产，路径使用 `Assets/TraitSO`。这样 Unity Inspector 里仍然可以看见每个羁绊，只是数据由表自动灌入。

`TraitSO` 继续作为运行时羁绊数据载体。后续应扩展它来保存 `UnionId`、`UnionDes`、档位列表和档位 `SkillID`，不要新建一套平行的羁绊资产系统。

羁绊图标建议放到独立目录 `Assets/Resources/Traits/Icons`，不要混在单位图标目录里。表格里写图标路径或图标名，由导入器在 Unity 里自动查找绑定。

同一个羁绊的 `UnionCounts` 不应重复。导入时如果发现重复档位，应报错并要求修表。

单位表里写了不存在的 `UnionId` 时，导入或运行时应警告，但不阻止单位生成；该缺失羁绊不生效。

子羁绊表里写了不存在的 `SkillID` 时，导入或进入战斗时应警告，但不阻止羁绊激活；缺失技能不执行。

新增 `TraitCSVImporter`，读取 `Trait.csv` 和 `TraitTier.csv`，生成或更新 `Assets/TraitSO/*.asset`。

## 激活规则

羁绊只统计已部署玩家单位。

备战席单位、敌人单位、召唤单位不参与羁绊人数统计。召唤单位可以吃到技能效果，但不会改变羁绊档位。

一个单位可以拥有多个羁绊，单位表 `UnionId` 字段继续支持多值写法，例如 `U1001;U2001`。

单位和羁绊的绑定通过单位表 `UnionId` 完成。生成单位时，`UnitDataManager` 读取 `UnitLogicDataSO.unionId`，按分隔符拆成多个 ID，再去匹配 `TraitSO.unionId`，最后把匹配到的羁绊写入单位的 `RuntimeTraits`。

同一个单位的不同星级只算一个羁绊人数。计数时使用单位的羁绊身份键，优先由 `ChessId` 去掉最后的星级数字得到，例如 `A1011`、`A1012`、`A1013` 都按 `A101` 计为同一个单位。

只取最高已激活档位。高档位会替换低档位；如果高档位需要保留低档位效果，需要在高档位的 `SkillID` 里把低档位技能也写进去。

`SkillID` 可以为空，用于暂时没有战斗效果的占位档位。导入时可以提示，但不应阻止使用。

## 技能结算

羁绊档位本身不直接实现护盾、伤害、召唤、复活或属性变化。羁绊档位只通过 `SkillID` 调用技能表，具体效果由 `SkillTable` 决定。

全体类羁绊档位技能每个档位只结算一次，不会因为有多个单位拥有该羁绊而重复释放。比如“全体友军护甲 +5%”只结算一次，每个目标单位按自己的护甲计算 5%。

局部一对多羁绊档位技能可以由每个拥有该羁绊的已部署单位分别发起。比如“每个重甲单位给周围 1 格友军加盾”，每个重甲单位都会以自己为中心各算一次，盾量可以使用该发起单位的属性。

是否属于全体类或局部一对多类，不在 `TraitTier.csv` 新增字段判断，而由 `SkillTable.TargetType` 决定。

不同羁绊给同一个 `SkillID` 时，不做全局去重，各自按自己的羁绊来源结算。否则两个不同羁绊碰巧用了同一个技能，会被错误吞掉。

同一个档位的 `SkillID` 内部如果重复写了同一个技能，需要去重，避免 `{S1;S1}` 放两次。

## 实现分层

`SynergyManager` 负责计算当前羁绊人数、最高激活档位和激活档位技能。它本来就在统计羁绊人数，所以羁绊状态应该从这里产出。

`SkillRuntimeManager` 负责执行羁绊档位技能。它需要提供专门入口，例如 `ApplyTraitTierSkills(activeTraitTiers)`，不要把羁绊档位技能伪装成单位自己的 `skillIds`。

全体类和局部一对多类的判断继续由 `SkillTable.TargetType` 决定，不在 `TraitTier.csv` 新增额外字段。

羁绊 UI 从 `SynergyManager` 读取一个整理好的羁绊展示模型。UI 不直接遍历单位列表或自己计算羁绊规则。

## UI 展示

常驻羁绊栏参考云顶之弈式展示：每条羁绊显示图标、羁绊名、当前人数和全部档位刻度，例如 `2 / 4 / 6`。

常驻栏不直接展开所有效果文本，避免战斗界面太挤。

玩家长按或按住某条羁绊栏目时，显示该羁绊详情。详情里先展示该羁绊的 `UnionDes`，再展示该羁绊所有档位的 `LevelDes`，而不是只显示当前档或下一档。

如果 `LevelDes` 和 `SkillID` 实际效果不一致，战斗逻辑以 `SkillID` 为准，UI 文案需要由策划修正。
